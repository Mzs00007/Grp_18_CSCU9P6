DESIGN ARCHITECTURE QUESTIONS:

Q1: "Why did you choose MVC architecture (Model-View-Controller)?"
CORRECT ANSWER:
  "MVC separates concerns:
   • MODEL (vcfs.models): Pure data entities (Candidate, Reservation, etc)
     Responsibility: Store state, validate rules, transition states
     Example: Reservation validates time windows don't overlap
   
   • CONTROLLER (vcfs.controllers): Application logic handlers
     Responsibility: React to user actions, update models
     Example: CandidateController handles 'book meeting' → creates Reservation
   
   • VIEW (vcfs.views): Java Swing GUI interfaces
     Responsibility: Display model state, capture user input
     Example: CandidateScreen displays available offers
   
   This separation means:
   ✓ Models are testable without UI (pure Java, no GUI)
   ✓ Multiple views can use same models (CLI + GUI + API)
   ✓ Easy to debug (problem in booking? Check Reservation model, not screens)
   ✓ Professional standard (used by Google, Netflix, etc)
   
   In VCFS: If a booking breaks, we check:
   1. Model logic (Reservation) - is state transition wrong?
   2. Controller (CandidateController) - is logic wrong?
   3. View (CandidateScreen) - is display wrong?
   Separation makes root cause obvious."

Q2: "How do your classes interact? Walk us through a candidate booking a meeting."
CORRECT ANSWER (With Pkg names):
  "The flow is:
  1. CANDIDATE CLICKS 'Book Meet' on CandidateScreen (VIEW)
  
  2. CandidateController.bookMeeting(candidate, offer) called (CONTROLLER)

Q3: "What design patterns did you use? Where?"
CORRECT ANSWER:
  "We used three professional patterns:
  
  1. OBSERVER PATTERN (Event notification)
     Location: vcfs.core.CareerFairSystem + AdminScreen
     Problem solved: When admin creates organization, AdminScreen needs to know
     Solution: CareerFairSystem notifies (fires event) → AdminScreen updates
     Code: AdminScreen addPropertyChangeListener('organizationCreated', ...)
     Benefit: Loose coupling—UI doesn't directly query every time, just listens
  
  2. SINGLETON PATTERN (One shared instance)
     Location: vcfs.core.CareerFairSystem
     Problem solved: System needs single source of truth for all bookings
     Solution: private CareerFairSystem only one instance, accessed via getInstance()
     Code: public static synchronized CareerFairSystem getInstance() { ... }
     Benefit: Thread-safe shared state, no duplicate data
  
  3. STATE PATTERN (Manage state transitions)
     Location: vcfs.models.booking.Reservation + MeetingSession
     States: CONFIRMED → IN_PROGRESS → CANCELLED (Reservation)
     Problem solved: Must prevent invalid transitions (can't go CANCELLED→CONFIRMED)
     Solution: Use enum for states, validate transitions in setState() method
     Code: if (currentState == CANCELLED) throw IllegalStateException()
     Benefit: Prevents bugs, business rules embedded in code"

Q4: "How do you prevent double-booking of recruiting slots?"
CORRECT ANSWER:
  "Triple validation:
  
  VALIDATION 1: In Reservation Model
  ────────────────────────────────
  When creating new Reservation:
  ```java
  public Reservation(Candidate candidate, Offer offer, 
                    LocalDateTime start, LocalDateTime end) {
      // Check: Does this candidate already have conflicting meeting?
      for (Reservation existing : candidate.getReservations()) {
          if (existing.timeWindowsOverlap(start, end)) {
              throw new IllegalArgumentException(
                  'Candidate already booked for this time!');
          }
      }
      // Check: Is this offer slot still available?
      if (offer.hasTooManyReservations()) {
          throw new IllegalArgumentException('Slot at capacity');
      }
      // If both pass, create reservation
      this.candidate = candidate;
      this.offer = offer;
      this.state = ReservationState.CONFIRMED;
  }
  ```
  
  VALIDATION 2: In Controller
  ──────────────────────────
  CandidateController double-checks before passing to model:
  ```java
  public Reservation bookMeeting(Candidate candidate, Offer offer) {
      // First check: Is the slot in the valid time window?
      if (!offer.getStartTime().isBefore(candidate.getPreferredWindow().getEnd())) {
          throw new BookingException('Time slot outside fair hours');
      }
      // Second check: Is the slot still open?
      if (offer.getAvailableSeats() <= 0) {
          throw new BookingException('No seats available');
      }
      // Now let model validate (it has all context)
      return careerFairSystem.createReservation(candidate, offer, ...);
  }
  ```
  
  VALIDATION 3: In Database (Mock)
  ───────────────────────────────
  Even if UI/controller bypass checks, system stores reservations atomically:
  Pseudo-code: On commit, database checks exist no conflicts, fails if found
  
  RESULT: Extremely hard to book twice—would need to bypass 3 layers!"

Q5: "How are meetings tracked and managed throughout their lifecycle?"
CORRECT ANSWER:
  "Meeting lifecycle is: Created → Started → Ended → Archived
  
  CLASS: MeetingSession (vcfs.models.booking.MeetingSession)
  Timeline:
  1. Creation: When Reservation transitions to IN_PROGRESS
     - MeetingSession created with reference to Reservation
     - Assigned VirtualRoom and Lobby
     - Start time recorded
  
  2. Active: Recruiter + Candidate enter virtual room
     - Each participant checked in
     - Attendance record started
     - Timer begins
  
  3. Completion: When timer expires or user ends meeting
     - End time recorded
     - Attendance records finalized
     - State changes: MeetingState.ENDED
  
  4. Archived: Session data stored for audit trail
     - Attendance report created
     - Fair statistics updated
  
  Storage: vcfs.models.audit.AttendanceRecord
  - Records what happened: who, when, duration, outcome
  - Used by admin to verify fair compliance
  - Prevents fraud (recruiter can't claim they met candidate if log shows no)"

Q6: "What was a design challenge you faced, and how did you solve it?"
GOOD ANSWER (Shows problem-solving):
  "Challenge: Encapsulation Violation
  Problem: Our model classes had all public fields, meaning any class could
  directly modify Reservation.state without validation. This violated business
  rules (can't go CANCELLED → CONFIRMED).
  
  What we discovered: With all public fields:
  ```java
  // WRONG - Anyone can do this anywhere:
  reservation.state = ReservationState.CANCELLED;  // No validation!
  reservation.state = ReservationState.CONFIRMED;  // Invalid transition!
  ```
  
  Solution: We refactored to use private fields + setter methods with validation:
  ```java
  // CORRECT - Only valid transitions allowed:
  public void setState(ReservationState newState) {
      if (!isValidTransition(state, newState)) {
          throw new IllegalStateException(
              'Cannot transition from ' + state + ' to ' + newState);
      }
      this.state = newState;  // Only valid transitions reach here
  }
  ```
  
  This fixed it: Business rules now enforced in code, not just documentation.
  
  Learning: Professional Java enforces encapsulation. We learned this COSTS
  refactor effort but SAVES debugging effort later."
