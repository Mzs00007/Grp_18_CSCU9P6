   Purpose: REFLECT on growth and share WISDOM for next time

   ✓ GOOD EXAMPLE (380 words):
   "I gained several insights this week:
   
   **INSIGHT 1: Test-Driven Thinking**
   Before this week, I thought testing was an afterthought—write code, then
   test it afterwards. This week showed me the opposite: tests should guide
   design. By writing test cases first (even pseudocode), I discovered that:
   - Lobby's add() doesn't specify what happens with duplicates
   - The API could be clearer (should add() return a boolean for success?)
   - The null handling contract isn't documented
   
   Going forward, I'll write test cases BEFORE finalizing APIs—it forces you
   to think about edge cases and contracts upfront.
   
   **INSIGHT 2: Test Readability Matters**
   My first version had test names like testAddMethod1(), testAddMethod2().
   When Mohamed reviewed, he asked: 'What's the difference between test 1 and 2?'
   I realized: test names should document BEHAVIOR, not be sequential numbers.
   I renamed to testAddValidCandidate(), testAddNullCandidate(), etc.
   Now anyone (including me in 6 months) can understand each test's purpose
   from the name alone. This is professional practice.
   
   **INSIGHT 3: Thread Safety Hidden Complexity**
   The Lobby class uses LinkedList<Candidate>. I initially assumed it was
   thread-safe because Java Collections.synchronizedList(). But learned:
   synchronization on the list doesn't sync access patterns like:
     1. Get queue
     2. Check if candidate in queue
     3. Add candidate
   If two threads do this simultaneously, duplicates possible. This required
   rethinking test assumptions—tests should verify thread-safety constraints,
   even if implementation delegates to external libraries.
   
   **INSIGHT 4: Test Coverage Numbers Are Misleading**
   '100% method coverage' sounds great, but I discovered it can be false security.
   I had 100% coverage yet initially missed:
   - Empty queue edge case (for listWaiting())
   - Removing last element edge case
   - Session persistence across operations
   These required BEHAVIOR-focused testing ("What should happen in this
   scenario?") not just CODE-focused testing ("Did I run this line?").
   
   **HOW I'LL APPLY THIS:**
   For next week's ReservationTest, I'll:
   1. Write test names that describe the BEHAVIOR, not test number
   2. Consider thread-safety implications before writing assertions
   3. Focus on scenarios (what users do) before code paths (what compiles)
   4. Review tests with a teammate before finalizing—fresh eyes catch confusion
   
   **CONFIDENCE GAINED:**
   I now feel confident that thorough testing catches bugs before production.
   The Lobby class is solid, and I understand WHY through writing these tests."

SITUATION:
With core Lobby implementation complete, our group needed comprehensive test
coverage to verify correctness before integration testing. The Lobby class is
critical for UC-4 (Candidate Joins Session Waiting Area). I took responsibility
for the Lobby test suite while [Name] tackled MeetingSession and [Name] worked
on Reservation. This work enables the integration phase and supports our 
screencast demonstration.

TASK:
Design and implement a complete JUnit test suite for the Lobby class covering
all 5 public methods, edge cases, and state management. The challenge was
ensuring FIFO queue ordering under various add/remove scenarios while verifying
that null inputs are properly rejected with exceptions. I also needed to test
the session binding mechanism and collection access patterns.

ACTION:
I performed the following steps over 5 hours total:

1. Code Analysis (1 hour)
   - Reviewed Lobby.java API and documentation
   - Identified dependent classes (Candidate, MeetingSession)
   - Mapped out all public methods and their responsibilities

2. Test Design (1 hour)
   - Created test plan: 20 test methods in 4 categories
   - Designed diverse test data (100+ candidate permutations)
   - Planned for edge cases: empty queue, null candidates, rapid operations

3. Implementation (2.5 hours)
   - Wrote LobbyTest.java: 450 lines of test code
   - Created 20 test methods with @DisplayName annotations
   - Added comprehensive javadoc explaining each test rationale

4. Debugging & Refinement (0.5 hours)
   - Fixed null pointer issues in test fixtures
   - Verified all 20 tests pass consistently
   - Ensured test isolation (no cross-test state pollution)

METRICS:
• Tests created: 20 JUnit test methods
• Code written: 450 lines (excluding comments)
• Time breakdown: Analysis 1h, Design 1h, Implementation 2.5h, Debug 0.5h
• Test coverage: 100% of public methods (5/5)
• Pass rate: 20/20 tests passing

EVIDENCE:

RESULTS:

✓ Initialization Tests (2)
  - Confirmed empty queue on construction

  - Verified remove() graceful handling
  - Confirmed null rejection with exceptions

LEARNING:
Key insights gained this week:

→ FIFO testing complexity: Queue order is fragile - easily lost with improper
