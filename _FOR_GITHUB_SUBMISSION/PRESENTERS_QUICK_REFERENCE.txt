─────────────────────────────────────────────────────────────────────

NARRATOR:
"Now here's where it gets interesting. Watch what happens when David books an 
interview. This demonstrates the OBSERVER PATTERN - the most important design

ACTIONS:
□ Click GREEN "BOOK" button on first available slot
□ Show success message: "✓ Interview booked successfully!"
□ Candidate screen shows booking in "My Schedule" tab

NARRATOR:
"David's schedule now shows his confirmed interview. But here's the magic..."

ACTIONS:
□ Bring up Recruiter window (Alice's portal) - make visible on main screen
□ Click "Schedule" tab in recruiter portal
□ Show Alice's schedule NOW SHOWS David's booking!

NARRATOR:
"✓ INSTANT SYNCHRONIZATION! Alice didn't refresh the page. She didn't click a 
button. The system automatically notified her that her 8:30 slot is now booked.

This is the Observer pattern in action:
1. David books a slot (event occurs)
2. CareerFairSystem broadcasts PropertyChangeEvent
3. Alice's screen is listening (PropertyChangeListener)
4. Her portal updates immediately with the booking

This works across THREE portals simultaneously - Candidate, Recruiter, AND Admin 
all see the same data in real-time.

Let me demonstrate time advancement..."

ACTIONS:
□ Click on SystemTimer window (lower right, or switch to it)
□ Click "Advance 30 Minutes" button
□ Watch both portal windows update automatically

NARRATOR:
"Notice - no refreshes needed. Alice's 8:30 slot now shows 'COMPLETED' in green, 
and her 9:00 slot becomes active. Real-time event-driven architecture working 
perfectly.

This is multithreaded, thread-safe programming at enterprise level."


⏱️  PART 6: ARCHITECTURE EXPLANATION (14:00 - 16:00) — 2 MINUTES
─────────────────────────────────────────────────────────────────

NARRATOR:
"Let me explain the technical foundation. When David booked that interview, here's 
exactly what happened behind the scenes..."

[If possible, show code on screen via IDE]

NARRATOR:
"1. CANDIDATE PORTAL (GUI): CandidateScreen captures 'Book' button click
2. CANDIDATE CONTROLLER: CandidateController validates the request
   - Is David logged in? ✓ Yes
   - Is the slot available? ✓ Yes
   - Is David within booking limits? ✓ Yes
3. MODEL (Backend): CareerFairSystem updates the database
   - Reserves the interview slot
   - Fires PropertyChangeEvent to all listeners
4. DATABASE: Uses thread-safe collections (ConcurrentHashMap)
   - No race conditions
   - No double-booking possible
   - Even if 100 candidates click simultaneously
5. OBSERVER NOTIFICATION: All three portal screens update
   - Recruiter sees the booking
   - Admin sees it in audit log
   - Candidate sees it in their schedule
6. ERROR HANDLING: If anything fails
   - Exception caught and logged
   - User sees friendly error message
   - System remains stable

This is production-grade code with comprehensive error handling, logging, and 
thread safety throughout."

ACTIONS:
□ If showing code, highlight:

   - Error handling (try-catch)
   - Logging calls with LogLevel
